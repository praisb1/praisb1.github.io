Proxmox Home Lab — Active Directory Build

Category: Home Lab Stack: Proxmox · Windows Server 2022 · Ubuntu Server Status: Active / Ongoing
Built a bare-metal home lab on a Ryzen 7 3700X workstation running Proxmox VE. Goal was to replicate a real enterprise environment — Windows Server with Active Directory, managed Windows 10 clients, and a Linux file server — to develop practical, job-ready IT skills.

// 02

Installing Proxmox VE

Proxmox VE is a Debian-based bare-metal hypervisor that lets you run both full VMs (KVM) and lightweight containers (LXC) from a web UI. It was the right choice here because it's free, enterprise-grade, and widely used in real IT environments.

01.

Download the Proxmox VE ISO

Grabbed the latest ISO from proxmox.com/downloads. At the time, this was Proxmox VE 8.x.

02.

Create a bootable USB with Rufus

First time making a bootable drive — used Rufus on Windows. Selected the ISO, left partition scheme as GPT for UEFI systems, and wrote it. Took a few minutes of confusion figuring out that Rufus needs to write in DD image mode for Proxmox to work correctly, not ISO mode.

03.

Boot from USB and install

Entered BIOS, set USB as first boot device, and followed the Proxmox graphical installer. Set the target disk to the 256 GB SSD, configured a static IP for the management interface, and set a root password. Installation takes around 5 minutes.

04.

Access the web UI

After reboot, navigated to https://[host-ip]:8006 in a browser on another machine. Logged in as root. The Proxmox dashboard was up and running — ready to create VMs.

TIP After install, Proxmox will warn about no valid subscription. You can dismiss this or add the free non-subscription repo in /etc/apt/sources.list.d/ to keep updates flowing without needing a paid license.

// 03

Ubuntu Server VM — Samba File Share

The first VM was an Ubuntu Server instance — familiar territory. The goal was to verify the lab networking was functional by standing up a Samba file share and accessing it from another machine on the network.

01.

Upload the Ubuntu Server ISO to Proxmox

In the Proxmox web UI, navigated to local storage → ISO Images → Upload. Uploaded the Ubuntu 22.04 LTS server ISO.

02.

Create and configure the VM

Created a new VM — allocated 2 cores, 4 GB RAM, 32 GB disk. Attached the Ubuntu ISO as the boot CD-ROM. Standard network bridge (vmbr0) for connectivity.

03.

Install Ubuntu and configure Samba

Standard Ubuntu Server install through the console. Once up, installed Samba with apt install samba, created a shared directory, and edited /etc/samba/smb.conf to define the share. Verified connectivity by mounting the share from a Windows machine on the network.

smb.conf — share definition CONF
[labshare] path = /srv/samba/labshare browseable = yes read only = no guest ok = no valid users = @sambausers

// 04

Windows Server 2022 — Active Directory

The main objective of the lab. Getting a legitimate Windows Server license turned out to be easier than expected — Microsoft's Azure Dev Tools for Education (accessible via a .edu email address) provides free, genuine product keys for Windows Server, Windows 10/11, and other Microsoft software for educational use.

TIP Log in to the Azure portal with your .edu email. Navigate past the initial popups to Azure Dev Tools for Education — you'll find legit product keys for Windows Server, Visual Studio, and more. Completely free for students.
01.

Create the Windows Server VM

Uploaded the Windows Server 2022 ISO to Proxmox. Created a new VM with 4 cores, 8 GB RAM, 60 GB disk. Chose the Server Core (no GUI) option during install to conserve resources — all management done via PowerShell and CLI.

02.

Install Active Directory Domain Services

After booting into Server Core, the OS drops into a numbered menu. Navigated to option 15 to drop into a full PowerShell session, then ran the following to install and configure AD DS:

PowerShell — AD DS installation POWERSHELL
# Install the AD DS role Install-WindowsFeature AD-Domain-Services -IncludeManagementTools # Promote to domain controller (creates new forest) Install-ADDSForest ` -DomainName "lab.local" ` -DomainNetbiosName "LAB" ` -InstallDns:$true ` -Force:$true # Server will reboot automatically after promotion
03.

Create domain user accounts

With AD DS running, used New-ADUser in PowerShell to create test accounts that could be used to log in from any domain-joined machine — simulating how a real enterprise environment manages identity centrally.

PowerShell — create a domain user POWERSHELL
New-ADUser ` -Name "John Doe" ` -GivenName "John" ` -Surname "Doe" ` -SamAccountName "jdoe" ` -UserPrincipalName "jdoe@lab.local" ` -AccountPassword (ConvertTo-SecureString "P@ssw0rd!" -AsPlainText -Force) ` -Enabled $true

// 05

Windows 10 Pro VMs — Domain Clients

Two Windows 10 Pro VMs were created to act as managed client machines. The Pro edition is required — Windows 10 Home cannot join an Active Directory domain.

NOTE When creating a Windows VM in Proxmox, the recommended disk controller is VirtIO SCSI for performance — but Windows doesn't include VirtIO drivers by default, which causes the installer to see no available disk. This was the main issue hit during this phase.
01.

Join the domain

After a clean Windows 10 install, navigated to Settings → System → About → Join a domain, entered lab.local, and authenticated with the domain admin credentials. After a reboot, the machine was domain-joined and could log in with any AD user account created on the server.

02.

Verify centralized login

Signed out, then logged back in as a domain user (LAB\jdoe). Credentials were authenticated against the Windows Server DC — exactly how enterprise workstations behave in the real world.

// 06

Troubleshooting — VirtIO Disk Issue

ISSUE When booting the Windows 10 installer in Proxmox, no storage device was visible even though a disk had been allocated. Root cause: Windows does not include VirtIO drivers out of the box, so it cannot see the virtual disk controller.

The fix requires temporarily switching the disk to a SATA controller, injecting VirtIO drivers into Windows via Safe Mode, then switching back to VirtIO SCSI for normal operation. Here's the exact procedure that worked:

Boot the VM using SATA controller initially and install VirtIO drivers for Windows from the VirtIO ISO (download from Fedora's VirtIO driver project).
Force Windows into Safe Mode by running bcdedit /set "{current}" safeboot minimal in an admin CMD, then shut down the VM.
In Proxmox, change drive C: to use SATA controller.
Add a CD-ROM device in Proxmox pointing to the VirtIO SCSI driver. No media needed — the device presence alone is the trigger.
Change the Proxmox boot order to boot from SATA.
Start the VM. Because Windows detects the CD-ROM device, it bakes the VirtIO drivers into the system during the Safe Mode boot.
Disable Safe Mode: bcdedit /deletevalue "{current}" safeboot and shut down.
Remove the CD-ROM device added in step 4 from Proxmox.
Change drive C: back to VirtIO SCSI controller in Proxmox.
Update the Proxmox boot order to boot from VirtIO SCSI. Start the VM — storage is now recognized correctly.
TIP The simpler prevention: when creating a new Windows VM in Proxmox, attach the VirtIO ISO as a second CD-ROM before the first boot and load the drivers during Windows Setup using "Load driver." Saves the whole Safe Mode process.
